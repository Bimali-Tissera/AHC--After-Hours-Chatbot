## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, input border-radius |
| sm | 8px | Label-to-input gaps, button internal padding (vertical) |
| md | 16px | Card padding, form field spacing, sidebar item padding |
| lg | 24px | Card outer margin, section headings margin-bottom |
| xl | 32px | Login card max-width gutter, main content padding |
| 2xl | 48px | Login page vertical centering breathing room |
| 3xl | 64px | Not used in Phase 1 |

Exceptions:
- Login card: `max-w-sm` (384px) — not a spacing token, a width constraint
- Input fields: `py-2 px-3` (8px/12px) — 12px is off-grid but is a Tailwind form input utility (`px-3`) and does not extend to other components
- Full-screen centering on login: `min-h-screen` — not a spacing token

**Source:** RESEARCH.md login page example uses `p-8` (32px) card padding, `mb-4`/`mb-6` (16px/24px) field gaps. Derived and extended.

## Typography

| Role | Size | Weight | Line Height | Tailwind Classes |
|------|------|--------|-------------|-----------------|
| Body | 14px | 400 (normal) | 1.5 | `text-sm font-normal leading-normal` |
| Label | 14px | 600 (semibold) | 1.25 | `text-sm font-semibold leading-tight` |
| Heading | 20px | 600 (semibold) | 1.2 | `text-xl font-semibold leading-snug` |
| Display | 28px | 600 (semibold) | 1.15 | `text-2xl font-semibold leading-none` — reserved for page titles |

**Weights used:** 2 weights — 400 (normal) for body copy and table content; 600 (semibold) for labels, headings, display titles, and buttons. The size difference between Display (28px) and Heading (20px) provides sufficient visual hierarchy without a separate bold weight.

**Source:** RESEARCH.md login page example uses `text-sm font-medium` for labels, `text-xl font-semibold` for the "Admin Login" heading, `text-sm font-medium` for the button. Collapsed to 2-weight contract per design system constraint.

## Layout Contracts

### Login Page (`/admin/login`)

**Primary focal point:** The centered white login card — eye travels directly to the "Sign In" button as the single action available on the page.

```
[min-h-screen bg-gray-50 flex items-center justify-center]
  └── [Card: bg-white rounded-lg shadow w-full max-w-sm p-8]
        ├── [Heading: text-xl font-semibold text-gray-900 mb-6]  "Admin Login"
        ├── [Error banner: text-sm text-red-600 mb-4]  (conditional)
        └── [Form: space-y-4]
              ├── Email field (label + input)
              ├── Password field (label + input)
              └── Submit button (full-width, bg-blue-600)
```

- No logo, no nav, no footer
- Card is vertically and horizontally centered
- No decorative elements

### Admin Dashboard Shell (`/admin/leads`)

**Primary focal point:** The leads table in the main content area — the sidebar orients, but the table occupies 80% of the viewport and is where the user's attention lands.

```
[min-h-screen bg-gray-50 flex]
  ├── [Sidebar: bg-white border-r border-gray-200 w-64 flex-shrink-0 p-4]
  │     ├── [Practice name: text-sm font-semibold text-gray-900 mb-6]
  │     ├── [Nav item — Leads: active state bg-blue-50 text-blue-600 border-l-2 border-blue-600]
  │     └── [Logout button: text-sm text-gray-500 hover:text-gray-900 mt-auto]
  └── [Main: flex-1 p-8]
        ├── [Page title: text-2xl font-semibold text-gray-900 mb-6]  "Leads"
        └── [Table area: bg-white rounded-lg border border-gray-200]
              ├── [Table header row: bg-gray-50 text-sm font-semibold text-gray-500]
              └── [Table body rows: text-sm font-normal text-gray-900 divide-y divide-gray-200]
```

**Demo-ready table columns (Phase 1 stub — no real data, but rendered with structure):**

| Column | Width | Content |
|--------|-------|---------|
| Name | 20% | Patient name |
| Phone | 15% | Phone number |
| Email | 20% | Email address |
| Trigger Question | 30% | What they asked |
| Date | 15% | Timestamp |

The Phase 1 stub renders this table with the empty state (no rows). Columns and headers are present so the layout looks complete for the demo.

## Accessibility Contract

- All form inputs have explicit `<label>` elements with matching `htmlFor`/`id` pairs
- Submit button uses native `<button>` with `formAction` — keyboard-accessible by default
- Error message is in the DOM (not just visually indicated) so screen readers detect it
- Color contrast: `gray-900` on `white` = 16:1 (AAA); `blue-600` on `white` = 4.5:1 (AA); `red-600` on `white` = 4.5:1 (AA)
- Focus ring: `focus:ring-2 focus:ring-blue-500` — visible keyboard focus indicator on all inputs
- No touch targets below 44px height — inputs are `py-2` (32px minimum with text); button is `py-2` — acceptable since this is a desktop-first admin tool
